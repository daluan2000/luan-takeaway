

# 高并发订单链路增强设计


# 3 增强后的订单核心链路

整体增强后链路如下：

```text id="7u9v1u"
下单请求
↓
幂等校验
↓
Redis Lua 扣减库存
↓
写入订单消息队列
↓
MQ消费者延迟聚合
↓
批量更新数据库
↓
延迟补偿校验
```

# 4 下单幂等控制

---

# 4.1 引入订单幂等 Token

下单前由服务生成：

Redis：

```text id="6zv3ov"
order:token:{userId}
```

示例：

```text id="53zc4f"
order:token:1001
```

值：

```text id="n07f63"
UUID
```

TTL：

```text id="wnt3wu"
120 秒
```

---

---

# 4.2 下单时 Lua 原子校验

提交订单时：

Lua 同时完成：

```text id="7ww4iw"
校验 token
删除 token
```

保证：

```text id="4j8n1i"
同一 token 只能成功一次
```

---

Lua 示例逻辑：

```lua id="mruwz1"
if redis.call('get', key) == token then
    redis.call('del', key)
    return 1
end
return 0
```

---

---

# 4.3 幂等控制价值

解决：

* 重复点击下单
* 请求重放
* 前端重试

避免重复订单进入库存链路。

# 6 MQ 延迟落库与批量合并写库（重点增强）

这一部分是最接近真实互联网订单写削峰的关键增强。

---

# 6.1 当前问题

当前每条订单消息立即消费：

```text id="jlwm0n"
一条消息 → 一次 SQL
```

高峰时数据库会出现大量碎片事务。

---

---

# 6.2 增加消费者端缓冲池

MQ 消费端不立即写库，而是先进入缓冲集合：

```text id="4r9t5w"
内存缓冲队列
```

例如：

```text id="kk9tnj"
List<OrderStockUpdate>
```

---

---

# 6.3 满足条件后批量落库

满足任一条件：

---

## 条件1：达到批量阈值

例如：

```text id="oq3c1n"
100 条消息
```

---

## 条件2：达到等待时间

例如：

```text id="ydb4dz"
200 ms
```

---

触发批量更新：

```text id="jlwm9z"
batch update stock
```

---

---

# 6.4 合并同菜品写操作

若缓冲期间出现：

```text id="d6jlwm"
dishId = 101 连续出现 20 次
```

先本地合并：

```text id="u3m2kn"
101 -> total -20
```

最终只生成：

```text id="xg0n9f"
一次库存更新 SQL
```

---

---

# 6.5 最终批量写库效果

原本：

```text id="85mn2w"
20次 update
```

变成：

```text id="w1a1i4"
1次 update
```

---

极大降低：

* MySQL IO
* 行锁竞争
* 事务数量

---

---

# 6.6 推荐批量 SQL 形式

例如：

```sql id="dcm1u1"
update wm_dish
set stock = stock - ?
where id = ?
```

批量执行：

```text id="5i3m0s"
batchUpdate
```

由 MyBatis 批处理即可。

# 8 延迟补偿机制

---

订单创建后发送延迟消息：

```text id="hgjr8w"
30分钟后检查订单状态
```

---

若订单异常：

例如：

* 下单成功但库存落库异常

则触发：

```text id="8rjlwm"
库存补偿恢复
```

---

形成：

```text id="jlwm6q"
Redis库存 ↔ DB库存 最终一致
```

这个“延迟补偿机制”的核心本质，确实可以理解成：利用延迟消息，对异步链路做一次兜底校验，确认 MQ 消息是否真正完成了业务闭环，检查最终业务状态是否达成

# 10 Future Work（更接近真实大规模订单系统）

---

## 10.1 批量写进一步升级为库存写聚合服务

当前批量聚合仍在消费者内部完成。

更大规模时通常拆为独立：

```text id="jlwmfw"
Inventory Aggregator Service
```

统一处理：

* 扣减合并
* 写入调度
* 异常恢复

---

---

## 10.2 更高吞吐 MQ 升级

更高并发场景通常采用：

Apache Kafka

适合：

* 更高消息吞吐
* 多消费链路并发

---

---

## 10.3 数据库分片

订单规模扩大后通常按：

* 用户
* 时间

分表：

```text id="jlwmk1"
order_202603
order_202604
```

降低单表压力。

---

---

## 10.4 库存服务独立化

当前库存仍属于 dish 服务。

真实系统通常拆出：

```text id="jlwmk2"
Inventory Service
```

统一负责库存生命周期。

---

---

## 10.5 网关侧下单限流

高峰期可在 Spring Cloud Gateway 增加：

```text id="jlwmk3"
商户维度限流
用户维度限流
```

提前削峰。
