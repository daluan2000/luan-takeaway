# 微服务如何以单体形式运行（Pig 项目）

## 1. 核心结论

在 Pig 中，“以单体形式运行微服务”本质是：
- 用 `pig-boot` 作为**单一启动器**；
- 通过 Maven 依赖把原本拆分的业务模块（认证、用户、代码生成、任务等）加载到**同一个 JVM 进程**里；
- 通过 Spring Boot 自动装配 + 组件扫描，把这些模块的 Bean 统一注册到一个 `ApplicationContext` 中；
- 对外只暴露一个 Web 入口（`/admin`）。

这不是把代码改成传统“大单体”，而是“保留模块边界、运行时合并进程”。

---

## 2. 代码层面是怎么做到的

### 2.1 单体启动入口
- 启动类：`pig-boot/src/main/java/com/pig4cloud/pig/PigBootApplication.java`
- 关键点：
  - `@SpringBootApplication`：触发自动装配与组件扫描；
  - `@EnablePigResourceServer`：启用资源服务鉴权能力；
  - `@EnablePigDoc(value = "admin", isMicro = false)`：文档按单体模式暴露。

### 2.2 通过依赖“聚合模块”
- 文件：`pig-boot/pom.xml`
- `pig-boot` 直接依赖：
  - 必备：`pig-auth`、`pig-upms-biz`
  - 可选：`pig-codegen`、`pig-quartz`

因为这些模块被打进同一个应用 classpath，Spring 会扫描并注册它们的 Controller/Service/Mapper/Config。

### 2.3 单体化配置开关
- 文件：`pig-boot/src/main/resources/application.yml`
- 关键配置：
  - `server.port: 9999`
  - `server.servlet.context-path: /admin`
  - `spring.cloud.nacos.config.enabled: false`
  - `spring.cloud.nacos.discovery.enabled: false`
  - `security.micro: false`

含义：单体模式下不依赖“服务发现 + 配置中心”做服务间路由，全部模块直接本地调用。

---

## 3. 请求链路如何变化

## 3.1 微服务模式（多进程）
1. 请求先到 `pig-gateway`；
2. 网关按路由转发到 `pig-auth`、`pig-upms-biz` 等独立服务；
3. 服务间走 HTTP/Feign。

## 3.2 单体模式（单进程）
1. 请求直接到 `pig-boot`（`/admin/**`）；
2. 由同一个进程中的对应 Controller 处理；
3. 模块间调用可退化为同进程 Bean 调用（无网络转发开销）。

你仍然能看到“auth/upms/quartz/codegen”这些模块的业务边界，但运行时不再是多个独立服务。

---

## 4. 为什么这种模式可行

- **包结构统一**：模块包名都在 `com.pig4cloud.pig...` 范围内，主启动类同根包可覆盖扫描；
- **公共能力沉淀在 pig-common**：安全、Feign、MyBatis、日志等能力可在单体和微服务两种模式复用；
- **配置分层清晰**：通过 `security.micro`、Nacos 开关等在运行时切换“微服务行为”与“单体行为”。

---

## 5. 与微服务模式的取舍

### 优点
- 启动和调试成本低（本地一个进程即可联调核心业务）；
- 无网关转发与跨服务网络调用，排障更直接；
- 适合开发期、PoC、小规模部署。

### 代价
- 失去服务独立伸缩与独立发布粒度；
- 单进程故障影响范围更大；
- 资源隔离能力弱于微服务部署。

---

## 6. 你可以怎么用

- 想快速本地联调：优先启动 `pig-boot`；
- 想验证完整微服务链路（网关、注册、配置、独立扩缩）：切换到 `pig-gateway + pig-auth + pig-upms...` 分布式启动。

一句话总结：**Pig 的“单体模式”是把微服务模块以依赖聚合方式合并到一个运行进程中，代码仍模块化，部署变单体化。**
