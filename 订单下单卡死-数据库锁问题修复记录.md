# 订单下单卡死 - 数据库锁问题修复记录

## 1. 问题背景
在 `pig-boot` 单体模式下，前端点击下单后长时间无响应（看起来像后端卡死）。

本次记录聚焦数据库锁：
- 什么操作触发了什么锁。
- 为什么会出现长时间等待。
- 修复方案如何避免等待和超卖。

## 2. 现象与关键报错

### 2.1 用户侧现象
- 前端下单接口请求长时间 pending。
- 即使只下单 1 个菜品，也可能出现明显等待。

### 2.2 后端关键日志
`pig-boot/logs/pig-boot/error.log` 中出现：

- `Lock wait timeout exceeded; try restarting transaction`
- 失败 SQL（摘录）：

```sql
UPDATE wm_dish
SET stock = stock - 3
WHERE del_flag='0'
  AND (id = ? AND merchant_user_id = ? AND sale_status = ? AND stock >= ?)
```

- 堆栈定位到：
  - `com.luan.takeaway.takeaway.dish.service.impl.WmDishServiceImpl.deductStock(...)`
  - `com.luan.takeaway.takeaway.order.service.impl.WmOrderServiceImpl.createOrder(...)`

## 3. 数据库锁机制说明（本问题核心）

## 3.1 触发了哪些锁

1. `UPDATE wm_dish ...` 触发行级排他锁（X Lock）
- InnoDB 对被更新的记录加 X 锁，直到事务提交/回滚才释放。
- 这是隐式锁，不需要在 SQL 里显式写 `FOR UPDATE`。

2. 外键检查引发父表记录锁参与
- `wm_order_item.dish_id -> wm_dish.id` 有外键（`fk_wm_order_item_dish`）。
- 插入 `wm_order_item` 时，数据库需要校验父表 `wm_dish` 记录存在并保持一致性。
- 在同事务中如果随后又去更新同一父记录，容易与并发事务形成锁等待链。

相关外键定义（`db/pig.sql`）：

```sql
ALTER TABLE `wm_order_item`
  ADD CONSTRAINT `fk_wm_order_item_dish` FOREIGN KEY (`dish_id`) REFERENCES `wm_dish` (`id`),
  ADD CONSTRAINT `fk_wm_order_item_order` FOREIGN KEY (`order_id`) REFERENCES `wm_order` (`id`);
```

## 3.2 为什么会“长时间等待”

满足以下条件时，会出现典型锁等待：

1. 多个事务几乎同时更新同一菜品库存。
2. 订单流程中先插明细再扣库存，导致外键检查与库存更新对同一 `wm_dish` 记录产生竞争。
3. InnoDB 默认锁等待超时较长（通常 50s），因此请求在超时前会持续等待。

所以用户体感是“卡死”，实际上是事务在等待行锁。

## 4. 根因链路（按调用顺序）

下单链路（简化）：
1. `WmOrderServiceImpl.createOrder` 创建订单。
2. 调用 `DishApi.deductStock` 扣减库存。
3. `WmDishServiceImpl.deductStock` 更新 `wm_dish.stock`。

历史问题点：
- 旧流程曾出现“先插 `wm_order_item`，后扣库存”的顺序，和外键锁机制叠加后更容易造成等待。
- 库存扣减最初为逐条更新，冲突时会放大等待时间。

## 5. 修复方案

## 5.1 调整下单顺序，降低外键锁冲突
文件：`pig-takeaway/pig-takeaway-order/pig-takeaway-order-biz/src/main/java/com/pig4cloud/pig/takeaway/order/service/impl/WmOrderServiceImpl.java`

调整后顺序：
1. `save(order)`
2. 先 `dishApi.deductStock(...)`
3. 再插入 `wm_order_item`

这样避免“明细外键校验先参与父表锁，再更新父表”的冲突放大。

## 5.2 库存扣减改为单 SQL 原子批量扣减
文件：`pig-takeaway/pig-takeaway-dish/pig-takeaway-dish-biz/src/main/java/com/pig4cloud/pig/takeaway/dish/service/impl/WmDishServiceImpl.java`

核心改动：
- 先合并同菜品数量（`TreeMap`，顺序稳定）。
- 使用单条 SQL 批量原子扣减（`CASE WHEN`）：

```sql
UPDATE wm_dish
SET stock = CASE id
  WHEN ? THEN stock - ?
  WHEN ? THEN stock - ?
  ...
  ELSE stock
END
WHERE del_flag='0'
  AND merchant_user_id = ?
  AND sale_status = '1'
  AND id IN (...)
  AND ((id=? AND stock>=?) OR (id=? AND stock>=?) ...)
```

判定规则：
- `updatedRows == 菜品种类数` 才算成功。
- 否则判定库存不足或状态变化，直接失败。

意义：
- 一次 SQL 内完成多菜品扣减，缩短事务持锁时间。
- 保留原子性，防止超卖。

## 5.3 缩短锁等待并增加短重试（防挂死）
同文件新增：
- 会话级设置：`SET innodb_lock_wait_timeout = 3`
- 轻量重试：`LOCK_RETRY_TIMES = 2`
- 捕获 `CannotAcquireLockException` 后短暂退避重试，最终返回明确业务错误：
  - `系统繁忙，请稍后重试下单`

意义：
- 即使遇到锁竞争，也能快速失败，不再让前端长时间等待。

## 6. 为什么“一个菜品也会卡”
即使只下 1 个菜，也可能卡：
- 并发请求都更新同一行 `wm_dish(id=...)`，X 锁互斥仍然成立。
- 事务内若存在外键关联写入与库存更新的时序不当，同样会放大等待。
- 本质是热点行竞争，不取决于菜品数量是否大于 1。

## 7. 验证与结果

已完成模块编译验证：
- `pig-takeaway-dish-biz` 编译通过
- `pig-takeaway-order-biz` 编译通过

修复后预期：
1. 下单请求不再长时间挂起。
2. 冲突场景下快速返回“系统繁忙/库存不足”，而不是等待到超时。
3. 不会引入超卖。

## 8. 运维与排查建议

1. 发布后务必重启 `pig-boot`，确保加载最新代码。
2. 若仍有偶发等待，优先排查热点菜品并发是否异常集中。
3. 建议保留以下排查命令模板：

```sql
SHOW FULL PROCESSLIST;
SELECT trx_id,trx_state,trx_started,trx_wait_started,trx_mysql_thread_id,trx_query
FROM information_schema.innodb_trx;
```

4. 若业务量继续上升，可进一步考虑：
- 下单削峰（限流/队列化）。
- 对热点菜品做更细粒度并发治理。

## 9. 本次结论
本次“后端卡死”本质是 InnoDB 行锁等待，不是 JVM 真正死锁。通过调整事务操作顺序、原子批量扣减、缩短锁等待与短重试，问题得到工程化修复。

# 一、InnoDB 行锁到底是什么

MySQL InnoDB 的核心规则：

> **只要 UPDATE / DELETE / SELECT FOR UPDATE，就会给记录加排他锁 (X Lock)**

排他锁规则：

```
事务A拿到锁
事务B必须等待
直到事务A提交
```

举例：

表 `wm_dish`

| id | stock |
| -- | ----- |
| 1  | 10    |

### 事务A

```sql
BEGIN;

UPDATE wm_dish
SET stock = stock - 1
WHERE id = 1;
```

此时：

```
id=1 这一行被加 X Lock
```

### 事务B

```sql
BEGIN;

UPDATE wm_dish
SET stock = stock - 1
WHERE id = 1;
```

B 会 **等待 A 释放锁**。

等待最长时间：

```
innodb_lock_wait_timeout
默认 50 秒
```

这就是你看到的：

```
Lock wait timeout exceeded
```

本质就是：

> **第二个事务在等锁**

---

# 二、你的 SQL 触发了什么锁

你的核心 SQL：

```sql
UPDATE wm_dish
SET stock = stock - 3
WHERE id = ?
```

执行流程：

```
1 MySQL 找到 id 对应记录
2 对这条记录加 X Lock
3 修改 stock
4 等事务提交才释放锁
```

所以：

```
每个订单都会锁 wm_dish 这行
```

如果多个订单买同一道菜：

```
全部竞争同一行锁
```

结构：

```
订单1  → lock dish(1)
订单2  → wait
订单3  → wait
订单4  → wait
```

---

# 三、外键为什么会加锁

你这里还有一个关键结构：

```
wm_order_item.dish_id → wm_dish.id
```

外键约束：

```sql
ALTER TABLE wm_order_item
ADD CONSTRAINT fk_wm_order_item_dish
FOREIGN KEY (dish_id) REFERENCES wm_dish(id)
```

当你执行：

```
INSERT wm_order_item
```

MySQL 必须保证：

```
dish_id 对应的 dish 存在
```

因此 InnoDB 会：

```
读取 wm_dish(id)
并对父表记录加共享锁 (S Lock)
```

为什么？

防止这种情况：

事务A：

```
INSERT wm_order_item(dish_id=1)
```

事务B：

```
DELETE wm_dish WHERE id=1
```

如果没有锁：

```
订单引用了一个被删除的菜
```

所以 InnoDB 必须锁父记录。

---

# 四、真正的问题：锁顺序冲突

旧流程：

```
1 插入 wm_order_item
2 更新 wm_dish.stock
```

数据库内部锁：

```
INSERT order_item
    ↓
S Lock wm_dish

UPDATE wm_dish
    ↓
X Lock wm_dish
```

注意：

```
S Lock → X Lock
```

这叫

```
锁升级
```

如果多个事务同时发生：

事务A：

```
S lock dish
等待 X lock
```

事务B：

```
S lock dish
等待 X lock
```

结构变成：

```
A 等 B
B 等 A
```

虽然 MySQL 会检测死锁，但很多时候变成 **长等待链**。

最终表现就是：

```
前端一直 pending
```

---

# 五、为什么一个菜也会卡

因为：

```
所有订单都更新同一行
```

假设 10 个用户同时下单：

```
dish_id = 1
```

锁队列：

```
Tx1 lock
Tx2 wait
Tx3 wait
Tx4 wait
Tx5 wait
```

如果事务执行时间是：

```
200ms
```

队列就变成：

```
Tx5 等 800ms
```

如果某个事务慢：

```
3s
```

那所有人都卡。

---

# 六、为什么调整顺序能解决

你现在顺序：

```
1 save order
2 扣库存
3 插 order_item
```

锁顺序：

```
UPDATE wm_dish
  ↓
X Lock

INSERT order_item
  ↓
S Lock
```

注意：

```
X Lock > S Lock
```

如果已经有 X Lock：

```
S Lock 不会阻塞
```

因为：

```
同一事务
```

数据库不会冲突。

所以：

```
不再需要锁升级
```

避免锁等待。

---

# 七、批量 SQL 为什么更好

旧逻辑：

```
for item:
   UPDATE dish
```

可能产生：

```
锁 → 释放 → 锁 → 释放
```

时间拉长。

新 SQL：

```
UPDATE wm_dish
SET stock = CASE ...
```

一次完成：

```
一次锁
一次更新
一次释放
```

事务时间更短。

锁竞争就更小。

---

# 八、为什么加 lock timeout + retry

你加了：

```
innodb_lock_wait_timeout = 3
```

意思：

```
等 3 秒就放弃
```

否则默认：

```
50 秒
```

用户体验：

```
前端卡 50 秒
```

现在：

```
3 秒失败
```

然后 retry：

```
重试 2 次
```

很多瞬时锁冲突就解决了。
