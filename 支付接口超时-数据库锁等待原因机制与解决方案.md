# 支付接口超时（`/pay/mock`）锁等待分析与解决方案

## 1. 现象与结论
- 前端调用：`http://localhost:8888/api/admin/pay/mock`
- 现象：前端长时间等待，最终超时或后端 500。
- 日志关键信息：
  - `OrderApi#markPaid(Long)` 调用报错
  - `MySQLTransactionRollbackException: Lock wait timeout exceeded; try restarting transaction`
- 结论：本次问题本质是 **MySQL InnoDB 行锁等待超时**，发生在更新 `wm_order` 支付状态时。

## 2. 这次为什么会发生

## 2.1 真实报错位置
从 `luan-boot/logs/luan-boot/error.log` 可见，超时出现在：
- `POST /admin/order/service/{orderId}/pay-success`（也就是 `OrderApi#markPaid`）

说明并不是 `/pay/mock` 控制器没进来，而是进来后在调用订单状态更新时卡住。

## 2.2 当前调用链（单体 pig-boot）
在 `WmOrderPayServiceImpl#mockPay` 中：
1. 查询订单（Feign 调 `orderApi.getById`）
2. 调 `orderApi.markPaid(orderId)` 更新订单状态
3. 写入 `wm_order_pay`
4. 调 `deliveryApi.createDeliveryOrder`

虽然是单体启动，但这里仍通过 Feign 发起 HTTP 调用回到同一个应用，形成“同库、多事务、同步串行调用链”。

## 2.3 触发锁等待的核心机制
InnoDB 机制要点：
- `UPDATE wm_order ... WHERE id = ?` 需要拿该行的排他锁（X 锁）。
- 如果该行已被其它事务持有不兼容锁（另一个更新/长事务未提交），当前事务会等待。
- 等待超过 `innodb_lock_wait_timeout` 就抛 `Lock wait timeout exceeded`。

本次支付流程中，`markPaid` 正是这个等待点。

## 2.4 这是不是“分布式事务”问题
严格来说：**不是 Seata/XA 这类全局分布式事务故障**。

但它有“分布式调用链”的形态：
- 服务 A（pay）在本地事务里同步调用服务 B（order）
- B 在同一个数据库上做更新
- 任一环节遇到行锁竞争，就会放大为整条链路超时

所以更准确描述是：
- **跨服务同步调用 + 同库事务竞争**导致的锁等待超时
- 不是“分布式事务协调器异常”

## 3. 为什么你调整顺序后有帮助
这次我们已将 `mockPay` 中顺序改为：
1. `markPaid`
2. `save(pay)`
3. `createDeliveryOrder`

这能减少以下风险：
- 把“最容易被阻塞的状态更新”前置，尽早失败或成功，避免后续动作先执行造成回滚链变长。
- 降低事务中不必要的前置写操作，缩短关键路径上的锁竞争窗口。

注意：这能显著缓解，但不是唯一根治手段。

## 4. 这次项目里还存在的放大因素
- 支付、接单、取消等都可能更新同一行 `wm_order`，并发点击会产生竞争。
- 单体模式下仍使用 Feign 走 HTTP 回调自身，链路更长、失败面更大。
- `mockPay` 事务里串行做多件事（更新订单、写支付、建配送单），任一环节慢都可能拖长事务。

## 5. 推荐的治理方案（按优先级）

## 5.1 必做：幂等与条件更新
- 支付幂等：同一订单已支付时直接返回成功（或明确提示），避免重复支付并发打同一行。
- 订单状态更新使用条件更新：
  - `UPDATE wm_order SET order_status='1', pay_time=NOW() WHERE id=? AND order_status='0'`
  - 受影响行数为 0 时按“状态已变化”处理，不进入长等待。

## 5.2 必做：状态机约束前后端一致
- 已下单（待支付）-> 可支付
- 已支付 -> 商家可接单
- 前端按钮和后端校验必须一致，避免无效请求并发冲突。

## 5.3 建议：缩短事务、拆分副作用
- 主交易只做“状态变更 + 支付记录”。
- 创建配送单改异步（事件/outbox）或事务后执行，避免把非关键动作放在同一事务里。

## 5.4 建议：单体模式减少自调用 Feign
- 在 `pig-boot` 单体场景，可优先本地 service 调用，减少 HTTP/Feign 回环。
- 微服务模式再走 Feign，保持兼容。

## 5.5 建议：数据库与运行参数
- 合理设置 `innodb_lock_wait_timeout`（不建议盲目调很大）。
- 重点是减少锁冲突，而不是延长等待时间。

## 6. 可直接执行的排查命令（下次复现时）
```sql
-- 查看当前锁等待（MySQL 8）
SELECT * FROM performance_schema.data_locks;
SELECT * FROM performance_schema.data_lock_waits;

-- 查看正在执行事务
SELECT * FROM information_schema.innodb_trx;

-- 查看当前会话与阻塞关系
SHOW PROCESSLIST;
```

## 7. 本次修复摘要
- 已将 `WmOrderPayServiceImpl#mockPay` 顺序调整为：先 `markPaid`，后 `save(pay)`。
- 已将服务名常量统一为 `TAKEAWAY_USER_SERVICE`（适配 merchant/customer/delivery 合并到 `pig-takeaway-user`）。
- 已将业务流程统一为：待支付可支付、已支付可接单（前后端一致）。

## 8. 后续建议
- 加一版“支付幂等 + 条件更新 + 配送单异步化”作为彻底治理版本。
- 给 `mockPay` 增加结构化日志（orderId、旧状态、新状态、耗时、失败阶段），便于快速定位下一次问题。
