# Pig 二次开发：扩展新业务模块指南（微服务 + 数据库）

本文基于当前仓库结构给出一套可直接执行的扩展方案，适用于在现有 `pig` 项目上新增一个业务域（示例名：`pig-order`）。

---

## 1. 目标与原则

- 保持现有分层风格：`聚合模块 + api + biz`。
- 优先复用已有能力：`pig-common-*`、`@EnablePigResourceServer`、`@HasPermission`。
- 先打通最小闭环（查询接口）再扩展复杂业务。

---

## 2. 推荐模块结构

新增一个业务域聚合模块：

```text
pig-order/
  pom.xml                  # 聚合 pom
  pig-order-api/
    pom.xml                # DTO/VO/Feign 接口
    src/main/java/...
  pig-order-biz/
    pom.xml                # 真正运行的微服务
    src/main/java/...
    src/main/resources/application.yml
```

参考现有实现：
- `pig-upms/pom.xml`
- `pig-upms/pig-upms-api/pom.xml`
- `pig-upms/pig-upms-biz/pom.xml`

---

## 3. 新建微服务步骤

## 3.1 注册模块到根聚合
在根 `pom.xml` 的 `<modules>` 新增：
- `pig-order`

## 3.2 编写 `pig-order/pom.xml`
- `packaging` 设为 `pom`
- `<modules>` 包含 `pig-order-api`、`pig-order-biz`
- `parent` 指向根工程 `pig`

## 3.3 编写 `pig-order-api/pom.xml`
建议依赖：
- `pig-common-core`
- `pig-common-feign`
- `pig-common-mybatis`

用于放：
- 公共 DTO/VO
- 对外 Feign 接口定义

## 3.4 编写 `pig-order-biz/pom.xml`
建议依赖：
- `pig-order-api`
- `pig-common-security`
- `pig-common-log`
- `pig-common-swagger`
- `pig-common-feign`
- `mybatis-plus-spring-boot3-starter`
- `mysql-connector-j`
- `spring-cloud-starter-alibaba-nacos-discovery`
- `spring-cloud-starter-alibaba-nacos-config`

## 3.5 新增启动类
在 `pig-order-biz` 新建启动类（如 `PigOrderApplication`），注解建议对齐：
- `@SpringBootApplication`
- `@EnableDiscoveryClient`
- `@EnablePigResourceServer`
- `@EnablePigFeignClients`
- `@EnablePigDoc(value = "order")`

可参考：
- `pig-upms/pig-upms-biz/src/main/java/com/pig4cloud/pig/admin/PigAdminApplication.java`

## 3.6 新增服务配置
在 `pig-order-biz/src/main/resources/application.yml` 配置：
- `server.port`
- `spring.application.name`
- `spring.cloud.nacos.discovery/config`
- `spring.config.import`（加载公共配置 + 服务配置）

可参考：
- `pig-upms/pig-upms-biz/src/main/resources/application.yml`

## 3.7 网关路由
在 Nacos 网关配置中增加路由：
- `Path=/order/**` -> `lb://pig-order-biz`

这样前端通过网关即可访问新服务。

---

## 4. 数据库扩展步骤

当前项目未内置 Flyway/Liquibase，建议采用“SQL 脚本 + 版本目录”管理。

## 4.1 业务表设计
- 新增业务表使用统一前缀（如 `order_*`）
- 必要索引、唯一约束、逻辑删除字段与审计字段保持现有风格

## 4.2 初始化脚本
建议新增目录：

```text
db/migrations/
  20260302_01_order_init.sql
  20260302_02_order_menu_perm.sql
```

并在需要一键初始化时合并到：
- `db/pig.sql`

## 4.3 权限菜单数据
接口用了 `@HasPermission("order_xxx")` 后，还需插入权限菜单数据，否则前端看不到按钮权限。

至少维护以下表（以实际表结构为准）：
- `sys_menu`（目录/菜单/按钮）
- 角色-菜单关联表（如 `sys_role_menu`）

可参考：
- `db/pig.sql` 中 `sys_menu` 的插入样例（如 `sys_user_add/sys_user_edit`）

## 4.4 Nacos 初始化（可选）
若你希望全新环境一键导入配置，同步维护：
- `db/pig_config.sql`

---

## 5. 代码落地清单（最小闭环）

在 `pig-order-biz` 内至少完成：

1. `entity`：如 `OrderEntity`
2. `mapper`：`OrderMapper`
3. `service`：`OrderService`
4. `controller`：`OrderController`
5. 权限注解：`@HasPermission("order_query")`
6. 一个查询接口：`GET /order/page`

在 `pig-order-api` 内补：
- 对外 DTO/VO
- 其他服务调用用的 Feign 接口（如需要）

---

## 6. 前端接入步骤（pig-ui）

1. 新增 API 封装（如 `src/api/order/index.ts`）
2. 新增页面（如 `src/views/order/index.vue`）
3. 新增路由与菜单配置
4. 按钮权限编码与后端 `@HasPermission` 保持一致（例如 `order_add`、`order_edit`）

---

## 7. 单体模式兼容（可选）

如果还需要 `pig-boot` 单体运行新模块：
- 在 `pig-boot/pom.xml` 增加 `pig-order-biz` 依赖

这样：
- 微服务模式：独立部署 `pig-order-biz`
- 单体模式：由 `pig-boot` 聚合启动

---

## 8. 推荐实施顺序

1. 建模块骨架（pom + 启动类 + yml）
2. 建表并打通一个查询接口
3. 接权限菜单与角色
4. 接前端页面
5. 补写导入初始化 SQL 和部署文档

---

## 9. 常见坑位

- 只加了 `@HasPermission`，没插权限码菜单数据，导致前端按钮不可见。
- 服务名、网关路由、前端请求前缀不一致。
- 新增模块没加入根 `pom.xml` 的 `<modules>`，导致根构建不包含该模块。
- 本地能跑、容器不能跑：Nacos 配置未导入或环境变量缺失。

---

## 10. 一句话总结

在 Pig 上做二次开发，最稳妥的方式是：**按 `api + biz` 新建独立业务域，复用统一安全与网关体系，同时把“数据表 + 权限菜单 + 路由配置”作为同一批变更一起交付**。
