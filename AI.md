

# AI 点餐助手混合智能架构设计

为提升点餐系统对自然语言需求的理解能力与推荐能力，在现有微服务体系基础上引入独立的 AI Service，将智能能力从传统业务服务中解耦，构建面向自然语言交互的混合检索推荐架构。

该架构采用 **大模型语义理解 + 结构化检索 + 语义知识检索 + 融合排序 + 候选选取 + 推荐解释生成** 的统一链路，使系统既能处理明确约束查询，又能支持复杂语义推荐，并保证推荐结果具备业务可控性与自然语言解释能力。

---

# 一、总体架构

系统在网关后新增独立智能服务：

```
用户自然语言输入
↓
Gateway
↓
AI Service
↓
Query Understanding
↓
Hybrid Retrieval Orchestration
↓
Fusion Ranking
↓
LLM Candidate Selection and Explaination
↓
Response Generation
↓
返回推荐结果
```

其中 AI Service 不直接维护业务主数据，而是作为智能编排层调用已有业务服务与知识服务：

* 菜品服务负责结构化数据查询
* 知识库负责语义知识检索
* 大模型负责语义理解、候选选取与结果解释

整体保持原有 Spring Boot + Spring Cloud 微服务边界稳定。

---

# 二、核心设计思想

系统采用 Hybrid Retrieval 统一检索策略，将结构化检索与语义检索纳入同一条推荐链路，由大模型承担高层语义决策，由规则系统承担稳定候选生成。

核心执行逻辑：

```text id="5s0g4r"
LLM 负责理解需求
结构系统负责精确过滤
知识系统负责语义补充
规则系统负责候选排序
LLM 负责最终选取与解释
```

即：

```text id="5nlc2b"
Natural Language Query
→ Structured Intent Parsing
→ Structured Retrieval + Semantic Retrieval
→ Fusion Ranking
→ Candidate Selection
→ Explanation Generation
```

这种方式兼顾：

* 结构约束精度
* 语义推荐灵活性
* 推荐结果稳定性
* 推荐解释可读性

# 三、知识建模设计

系统为菜品建立独立知识层，将业务主数据与推荐知识解耦，形成可供结构检索、语义召回与解释生成共同使用的统一知识表示。

知识文档与业务菜品一一关联，由 AI Service 在不同阶段按需调用不同知识板块。

每一个菜品对应一个知识实体`DishKnowledgeDoc`，菜品的知识实体数据存放在数据库中，在菜品服务里实现知识实体的crud功能。

知识实体内按职责划分为三个板块，服务于不同阶段的智能处理任务。

## 结构化知识板块

承担精确约束表达职责，用于支持业务条件过滤与结构化意图解析。

特点：
* 字段边界明确
* 适合规则判断
* 可直接映射业务接口参数

主要用途：
* Query Understanding 阶段用于约束抽取
* Structured Retrieval 阶段用于业务检索执行
---

该板块强调可控性与稳定性，是结构检索的核心输入来源。

## 语义知识板块

承担推荐语义表达职责，用于描述菜品适用场景、偏好语义与语义关联特征。

特点：
* 支持语义近似表达
* 支持 embedding 构建
* 支持相似召回

主要用途：
* Query Rewriting 阶段辅助生成语义检索表达
* Semantic Retrieval 阶段参与向量召回
* Candidate Selection 阶段辅助候选语义比较

该板块用于补充结构字段无法直接表达的推荐语义。

## 描述知识板块

承担推荐解释职责，用于组织推荐理由与自然语言输出。

特点：
* 可读性优先
* 不参与过滤
* 不参与精确匹配

主要用途：
* Candidate Selection 阶段辅助候选理解
* Response Generation 阶段生成推荐解释

该板块主要服务于推荐结果的自然语言表达。


# 四、知识数据自动生成

AI-service实现基于LLM的知识数据自动生成功能，根据查询到的菜品基本信息，菜品名称、描述、价格等，由LLM自动生成结构化的知识实体数据。

当商家创建菜品或修改菜品时，前端页面应有选项，选择是否自动生成知识实体。然后后端菜品服务，使用rabbitmq，异步调研AI-service，获取并存储生成的知识实体。

在查询菜品列表时，知识实体并不一并返回。前端增加一列，知识库操作，商家可以点击查询该菜品的知识库，查询成功后可以手动编辑。


# 五、查询处理流程

系统采用分阶段智能处理机制。



## 1. Query Understanding

首先由大模型对用户自然语言进行统一语义解析，将输入拆解为两类信息：

* 结构约束意图
* 语义推荐意图

这一阶段本质上完成：

```
自然语言 → 可检索意图表示
```

其中：

* 结构约束意图用于后续业务过滤
* 语义推荐意图用于后续知识召回

此阶段主要依赖结构化知识板块完成约束理解。



## 2. Structured Retrieval

结构化检索由业务服务完成。

AI Service 根据解析结果调用已有业务接口，对菜品进行条件过滤。

执行方式：

```
AI Service → dish-service
```

完成：

* 条件筛选
* 范围过滤
* 分类匹配
* 基础候选集生成

该阶段保证推荐结果满足业务约束与实时数据一致性。



## 3. Semantic Retrieval

对于无法由结构字段直接表达的需求，引入语义知识检索机制。

系统通过向量化查询表达进入语义召回流程：

```
Query Embedding
↓
Vector Search
↓
Knowledge Recall
```

完成：

* 场景相关召回
* 语义相似召回
* 饮食知识补充

该阶段主要依赖语义知识板块完成召回。



# 六、融合排序机制

结构检索结果与语义召回结果首先进入统一排序阶段，由规则系统生成稳定候选集合。

排序综合考虑：

* 结构匹配程度
* 语义相关度
* 业务优先级

形成统一排序表达：

```
Structured Score + Semantic Score + Business Score
```

排序结果形成：

```
Top-K Candidate Set
```

该阶段目标是生成高质量候选空间，而非直接确定最终推荐结果。



# 七、候选结果选取与解释

在规则排序完成后，由大模型对候选集合进行最终语义判断。

大模型结合当前用户表达语境，从排序靠前的候选集中选取最适合当前场景的若干结果。

主要完成：
* 候选间语义比较
* 场景适配判断
* 最终推荐结果选取
* 选取理由解释

执行逻辑：

```
Rule-based Ranking → LLM Selection and Explaination
```

该阶段主要依赖：
* 结构化知识板块
* 语义知识板块
* 描述知识板块

通过有限候选空间中的高层语义推理，提升推荐灵活性。
