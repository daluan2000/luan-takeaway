把 AI 点餐助手设计成一个**独立的智能服务层（AI Service）**，挂在现有业务链路前面，而不是直接侵入订单服务。这样既保持你现有 Spring Boot + Spring Cloud 微服务结构稳定，又方便后续替换模型或增加推荐能力。整体上建议采用“两层决策 + 两类执行”的方式：第一层由大模型负责理解用户意图，第二层根据意图进入结构化工具调用或知识增强推荐。

---

### 一、整体架构（建议直接新增 `luan-ai` 服务） 🤖

在现有网关后新增一个 AI 服务：

```text
用户（前端聊天输入）
   ↓
Gateway
   ↓
luan-ai
   ↓
LLM 意图识别层
   ↓
├── Tool Calling（结构化查询）
└── RAG（语义推荐）
   ↓
返回推荐结果
   ↓
前端展示 + 一键加入购物车
```

这个服务建议独立成一个微服务：

```text
luan-ai
```

它不直接持有业务数据，而是调用你已有服务：

* 调用 `luan-takeaway-dish` 获取菜品数据
* 调用 `luan-takeaway-user` 获取用户历史偏好（可选）
* 调用 `luan-order` 获取历史订单（可选）

这样 AI 服务本质上只是一个“智能编排层”。

---

### 二、两层 AI 决策逻辑（这是核心） 🧠

#### 第一类：结构化需求 → Tool Calling

适合用户表达明确：

* 30元以内的麻辣烫
* 不要辣
* 两个人吃
* 热量低一点

此时 LLM 不直接生成答案，而是先输出结构化参数：

```json
{
  "category": "麻辣烫",
  "priceMax": 30,
  "spicy": false,
  "people": 2
}
```

然后 AI 服务调用你已有菜品接口：

```text
dish-service /search
```

再由 LLM 做结果整理：

```text
推荐3个符合条件的菜品，并解释原因
```

这个模式本质是：

```text
LLM负责理解
业务系统负责查数据
```

非常适合你现有微服务体系，因为数据库仍由业务服务掌控。

---

#### 第二类：模糊需求 → RAG 推荐

适合用户表达模糊：

* 今天胃不舒服吃什么
* 最近有点上火
* 晚上不想吃太油

这里不能直接查数据库，因为用户没有给出明确字段。

所以先做知识增强：

```text
用户问题
↓
向量检索（饮食知识 + 菜品标签）
↓
召回相关知识
↓
LLM生成推荐
```

知识库建议包含两类：

---

**① 菜品知识**

例如：

```text
皮蛋瘦肉粥：清淡、易消化、低刺激
番茄鸡蛋面：低油、适合胃不适
```

---

**② 饮食建议知识**

例如：

```text
胃不舒服避免：
辛辣
油炸
冰冷
```

---

最终 LLM 输出：

```text
建议吃清淡温热食物，可选择皮蛋瘦肉粥、番茄鸡蛋面。
避免麻辣和油炸。
```

然后再关联你自己的菜品数据：

```text
当前附近商家中有：
XX店番茄鸡蛋面
XX店南瓜粥
```

这样推荐就不是空泛的，而是落到你的业务数据上。

### 三、技术栈建议（与你当前系统兼容） ⚙️

后端建议：

* LangChain4j（非常适合你现有 Java 技术栈）
* 本地模型或远程模型：

  * Ollama（本地部署）
  * 或云端 API

向量检索建议：

* Redis 向量能力（如果继续沿用 Redis）
* 或 Milvus

因为你已有 Redis，轻量版可以直接：

```text
Redis + embedding缓存 + 小规模检索
```

---

工具调用建议：

AI 服务内部定义：

```java
DishSearchTool
MerchantSearchTool
OrderHistoryTool
```

由 LangChain4j 自动映射成 Tool Calling。
