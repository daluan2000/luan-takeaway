# Luan Takeaway 读链路增强设计文档

---

# 1 设计目标

当前系统已具备基础 Redis 缓存能力，并支持根据用户经纬度查询附近商户。

随着业务复杂度提升，需要进一步增强两个高频读链路：

1. **热点业务数据缓存能力增强**
2. **空间范围查询能力增强**

设计目标：

* 更接近真实外卖系统核心读链路设计
* 保持现有项目结构稳定
* 控制实现复杂度，适合个人项目逐步演进
* 体现真实互联网系统中的热点识别与空间索引思想

---

# 2 当前系统存在的问题

---

## 2.1 热点缓存问题

当前缓存机制主要解决：

* 缓存穿透
* 缓存击穿
* 缓存雪崩

但仍存在：

### 热点数据无法动态识别

当前所有数据采用统一 TTL：

```text
dish:123 ttl=5min
shop:88 ttl=5min
```

问题：

* 高频访问数据无法延长缓存生命周期
* 短时热点会频繁回源数据库
* 热门商家/热门菜品在高并发下仍可能产生 Redis 压力

---

## 2.2 空间查询问题

当前商户查询逻辑：

```text
根据用户经纬度
↓
计算经纬度矩形范围
↓
MySQL 经纬度范围查询
```

存在问题：

### 矩形范围过滤存在无效数据

因为：

```text
矩形区域 ≠ 实际距离范围
```

导致：

* 查出大量实际距离较远的商户
* 后续 Java 层过滤成本较高

### 查询无法缓存复用

当前查询结果强依赖用户精确经纬度：

```text
用户A位置 ≠ 用户B位置
```

导致：

* 相邻用户查询重复度高
* Redis 难以复用缓存结果

# 3 热点缓存增强设计

---

# 3.1 总体思路

在现有缓存组件基础上增加：

```text
访问统计 → 热点识别 → 热点缓存升温
```

整体链路：

```text
请求访问数据
↓
访问计数
↓
判断是否热点
↓
热点数据提升缓存等级
```

---

# 3.2 热点识别策略

采用轻量级时间窗口统计：

Redis Key：

```text
hot:counter:dish:{id}
hot:counter:shop:{id}
```

示例：

```text
hot:counter:dish:1001
```

TTL：

```text
60秒
```

每次访问：

```text
INCR key
```

达到阈值：

例如：

```text
60秒内 >= 100 次
```

判定热点：

加入热点集合：

```text
hot:set:dish
hot:set:shop
```

---

# 3.3 热点缓存策略

普通数据：

```text
TTL = 5 min
```

热点数据：

```text
TTL = 30 min
```

缓存访问逻辑变为：

```text
if 热点:
    使用热点TTL
else:
    使用普通TTL
```

---

# 3.4 热点缓存接口设计

新增统一组件：

```text
HotKeyManager
```

职责：

* 访问计数
* 热点判断
* 热点状态查询

核心接口：

```java
boolean isHotDish(Long dishId);

void recordDishAccess(Long dishId);
```

---

# 3.5 与现有 RedisSafeCacheService 集成

现有：

```text
RedisSafeCacheService
```

增强为：

```text
查询缓存前判断热点等级
```

缓存 TTL 动态决定：

```java
Duration ttl = hotKeyManager.isHotDish(id)
        ? Duration.ofMinutes(30)
        : Duration.ofMinutes(5);
```

---

# 3.6 当前阶段不引入的复杂机制

暂不实现：

* JVM 本地缓存（Caffeine）
* 热点异步预热
* 多级热点分层

原因：

当前项目规模下 Redis 已足够支撑。

# 4 空间查询增强设计

---

# 4.1 总体思路

将当前：

```text
经纬度范围查询
```

升级为：

```text
GeoHash 网格查询 + Redis 网格缓存
```

---

# 4.2 GeoHash 引入原因

GeoHash 将经纬度编码为固定网格：

例如：

```text
wx4g0e
```

同一区域用户会落入相同网格。

优势：

* 相邻用户共享查询缓存
* 可建立稳定缓存 key
* MySQL 查询更适合索引优化

---

# 4.3 数据库字段设计

商户表新增字段：

```sql
geohash_6 varchar(16)
```

索引：

```sql
index idx_geohash_6(geohash_6)
```

商户写入或更新位置时同步生成：

```text
经纬度 → geohash
```

---

# 4.4 查询流程

用户进入：

```text
用户经纬度
↓
计算 geohash
↓
获取当前格 + 邻近格
↓
查询 Redis
↓
未命中则 MySQL 回源
↓
距离精过滤
↓
返回商户列表
```

---

# 4.5 Redis 缓存设计

缓存 key：

```text
geo:shop:{geohash}
```

示例：

```text
geo:shop:wx4g0e
```

缓存内容：

```text
商户ID列表
```

TTL：

```text
5 min
```

---

# 4.6 查询缓存逻辑

先查：

```text
当前格子缓存
```

若 miss：

MySQL：

```sql
where geohash_6 in (...)
```

查出后写 Redis。

---

# 4.7 精确距离过滤

GeoHash 只是粗过滤。

仍需二次计算真实距离：

采用：

Haversine formula

最终保留：

```text
3km 内商户
```

---

# 4.8 当前阶段不引入复杂空间索引

暂不实现：

* Redis GEO
* Elasticsearch geo_distance
* 多层 GeoHash 精度切换

原因：

当前 MySQL + Redis 足够支撑项目规模。

# 10 Future Work（面向更接近真实大规模系统的后续演进）

当前系统已完成热点缓存识别与基于 GeoHash 的空间查询优化，能够较好覆盖个人项目阶段的核心高频读链路需求。但若进一步向真实大规模外卖平台靠近，在高并发、超大数据规模和多区域部署场景下，仍有若干方向可以继续演进。

---

## 10.1 热点缓存进一步演进

当前热点识别采用 Redis 时间窗口计数方式，适合中小规模服务部署。在更高并发场景下，真实系统通常会进一步引入：

### JVM 本地一级缓存（L1 Cache）

形成：

```text
Request
↓
JVM Local Cache
↓
Redis
↓
MySQL
```

常见实现是引入 Caffeine 作为进程内热点缓存，用于拦截极高频热点请求，降低 Redis 压力。

适用对象：

* 首页热门商户
* 高频菜品详情
* 热门推荐列表

---

### 热点异步预热机制

当前热点数据仍依赖请求触发缓存写入。

真实系统通常会增加：

```text
热点识别
↓
后台异步预热
↓
提前刷新热点缓存
```

即在热点即将过期前由后台任务提前刷新，而不是等待用户请求触发回源。

这样可进一步降低热点失效瞬间的数据库冲击。

---

### 热点动态降温与分层 TTL

当前热点数据仅采用统一长 TTL。

更完整的做法通常是：

* 短时热点（午餐时段）
* 长周期热点（常驻热门店铺）

分别使用不同缓存策略：

```text
短热点：TTL 10 min
长热点：TTL 1 h
```

使热点缓存更贴近真实访问规律。



## 10.2 空间查询进一步演进

当前空间查询采用：

```text
GeoHash + MySQL + Redis Cell Cache
```

已适合中等规模数据。

若进一步扩大商户规模，真实系统通常会继续演进为：

---

### 多级 GeoHash 精度控制

当前统一使用固定长度 GeoHash。

更接近真实系统时通常会根据区域密度动态调整：

* 商圈区域使用更细网格
* 郊区使用更粗网格

即：

```text
高密度区域 → geohash_6
低密度区域 → geohash_5
```

从而平衡：

* 查询精度
* 缓存复用率
* Redis Key 数量

---

### 空间索引服务独立化

随着位置查询频率继续提升，可将空间查询逻辑独立为专门服务：

```text
Geo Query Service
```

专门负责：

* GeoHash 编码
* 邻格扩展
* 距离过滤
* 商户区域缓存

避免业务服务中空间逻辑持续膨胀。

---

### 更高性能空间引擎接入

在更大规模场景下，常见方案包括：

* Elasticsearch 的 geo_distance 查询
* Redis GEO 索引能力

适合：

* 商户规模进一步扩大
* 支持复杂排序（距离 + 热度 + 配送能力）
